# Security Specification

## Data Invariants
- A ticket must belong to an existing customer.
- Insights are internal data generated by the system (or in this demo, calculated per user).
- Only authenticated admins (all users in this demo) can read/write data.

## The Dirty Dozen Payloads
1. Create customer with 2MB name string.
2. Create ticket for non-existent customerId.
3. Update customer health score directly (should be restricted if possible, but in this SPA it's fine for now).
4. Update createdAt/lastUpdated with client time instead of server time.
5. Create customer with missing required fields.
6. Inject script tags into description field.
7. Attempt to delete a customer without rights.
8. Update ticket severity to an invalid enum value.
9. Mass-read customers without authentication.
10. Spoof userId in a request.
11. Inject 10,000 items into deviceInventory array (if it were an array, but we use counts).
12. Update a resolved ticket to "Open" after it's been "Closed" (terminal state check).

## Strategy
- Use `isValidId` for all path variables.
- Use `isValidCustomer`, `isValidTicket`, `isValidInsight` helpers.
- Enforce `request.time` for timestamps.
- Terminal state check for tickets.
